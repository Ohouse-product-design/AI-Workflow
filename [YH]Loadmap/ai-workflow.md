### 1-1. Data-insight `P1` `Quick Win`

**목표**
디자이너가 직접 SQL을 작성하거나 대시보드를 뒤지지 않고, 자연어로 핵심 지표를 즉시 조회할 수 있게 한다.

**현재 문제**
- 핵심 지표(전환율, GMV, DAU 등) 확인에 Redash/Athena 직접 접속 필요
- 디자이너가 데이터를 보려면 DA에 요청하거나 직접 쿼리 작성 → 2~4시간 소요
- 데이터 기반 의사결정이 느려지고, 감에 의존하는 디자인 판단 발생

**솔루션**
- `/query` 스킬로 자연어 질문 → Athena/Redash MCP가 쿼리 생성 및 실행
- 결과를 Slack 채널에 요약 카드로 자동 공유
- 자주 쓰는 쿼리는 템플릿으로 저장

**솔루션 타입:** `MCP · Athena` `MCP · Redash` `Skill · /query`

**임팩트**
- 데이터 조회 시간: 2~4h → 5분 이내
- DA 의존도 감소, 디자이너 자율 데이터 접근

**기대효과**
- 디자인 리뷰 시 데이터 근거를 바로 제시 가능
- "이 수치 확인해주세요" 요청 대기 시간 제거
- 이미 MCP 인프라 구축 완료 → Quick Win

**선행 조건:** Athena MCP, Redash MCP 세팅 완료

### 1-2. VOC-collector `P2`

**목표**
Slack, CS 채널 등에 흩어진 사용자 피드백을 자동으로 수집 · 분류하여 디자인 인사이트로 전환한다.

**현재 문제**
- VOC가 Slack 여러 채널에 분산 → 수동으로 모아야 함
- 수집해도 카테고리 분류, 감정 분석 등 정리에 시간 소요
- 중요한 피드백이 묻히거나 누락되는 경우 빈번

**솔루션**
- Agent가 지정된 Slack 채널을 주기적으로 스캔
- VOC를 자동 카테고리 분류(UX 이슈 / 기능 요청 / 버그 / 칭찬)
- 감정 분석(긍정/부정/중립) 태깅
- 정리된 결과를 Notion DB에 자동 적재

**솔루션 타입:** `Agent · 자동수집` `MCP · Slack` `MCP · Notion`

**임팩트**
- VOC 수집 · 정리 시간: 주 3~4h → 자동화
- 누락 VOC 감소

**기대효과**
- 디자인 방향 설정 시 실제 사용자 목소리 기반 의사결정
- 반복 이슈 패턴 조기 발견
- 리서치 시작 전 VOC 현황 즉시 파악

**선행 조건:** Slack MCP, Notion MCP 세팅

### 1-3. Usecase-mapper `P2`

**목표**
현황 데이터와 사용자 흐름을 분석하여 유즈케이스 · 이탈 포인트를 자동으로 정리한다.

**현재 문제**
- 유즈케이스 정리가 디자이너 개인 역량에 의존
- 사용자 플로우 맵핑에 상당한 시간 소요
- 아웃포인트 · 이탈 구간 식별이 주관적

**솔루션**
- 기존 플로우 데이터 + 로그 데이터를 입력으로 받아 분석
- 주요 유즈케이스 자동 도출 및 정리
- 이탈 포인트 · 아웃포인트 시각화
- 시나리오 기반 플로우 자동 생성

**솔루션 타입:** `Skill · 플로우 분석` `Agent · 시나리오 생성`

**임팩트**
- 유즈케이스 정리 시간: 1~2일 → 2~3시간
- 이탈 포인트 식별 정확도 향상

**기대효과**
- 디자인 시작 전 사용자 여정 전체 맵을 빠르게 확보
- 데이터 기반 이탈 구간 우선순위 도출
- 팀 내 유즈케이스 정리 포맷 통일

**선행 조건:** 기존 플로우 데이터 정비 선행

### 2-1. Design-draft `P0`

**목표**
PRD와 ODS 컴포넌트를 기반으로 HTML 아이데이션부터 결론안까지 자동 생성하여, 디자인 초안 작업 시간을 획기적으로 줄인다.

**현재 문제**
- PRD 읽고 → 레퍼런스 찾고 → 와이어프레임 → 시안 과정이 3~8일 소요
- 초안 수준에서 여러 번 이터레이션 → 결론안까지 시간 소모
- ODS 컴포넌트를 매번 수동으로 찾아 조합

**솔루션**
- PRD + ODS 컴포넌트 레퍼런스를 Claude Code에 입력
- HTML 기반으로 여러 방향의 디자인 시안 자동 생성
- 초안을 넘어 결론안 수준까지 AI가 제안
- 확정된 시안을 Figma에 반영

**솔루션 타입:** `Skill · 결론안 생성` `Figma · ODS 컴포넌트` `Agent · Claude Code`

**임팩트**
- 디자인 초안 시간: 3~8일 → 1~2일 (시간 절약 61%+)
- 이터레이션 횟수 감소

**기대효과**
- PO/기획과의 디자인 논의를 실물 기반으로 빠르게 시작
- 여러 방향성을 동시에 탐색 가능 (A/B/C안 병렬 생성)
- 스텔라: "HTML 다 만들어두고 아이데이션해달라고 한다" — 이미 검증된 방식
- 잭: 초안보다 결론안까지 가는 것이 핵심

**선행 조건:** ODS 컴포넌트 정비, Claude Code 연동

### 2-2. Reference&Pattern-analyzer `P1`

**목표**
경쟁사 · 글로벌 레퍼런스 수집과 UI 패턴 분석을 하나의 워크플로우로 통합하여, 디자인 리서치 시간을 단축한다.

**현재 문제**
- 레퍼런스 수집(웹 서핑, 스크린샷, 정리)에 반나절~1일 소요
- 수집한 레퍼런스에서 패턴을 도출하는 것은 또 별도 작업
- Google Drive/Figma에 아카이브가 체계적이지 않음

**솔루션**
- Agent가 키워드/도메인 기반으로 경쟁사 · 글로벌 서비스 자동 탐색
- 주요 화면 캡처 및 UI 패턴 자동 분석 (레이아웃, 컴포넌트 사용, 인터랙션)
- STACK 컴포넌트 기준으로 패턴 매칭
- Figma 레퍼런스 페이지에 자동 정리 · Google Drive 아카이브

**솔루션 타입:** `Agent · 웹 리서치+패턴 분석` `Figma · UI 정리`

**임팩트**
- 레퍼런스 수집+분석: 4~8h → 1~2h
- 패턴 도출의 객관성 향상

**기대효과**
- 디자인 의사결정 시 "다른 서비스는 이렇게 한다"를 데이터로 뒷받침
- 팀 전체가 공유하는 레퍼런스 아카이브 구축
- 기존 Reference-scout + UI-pattern-analyzer 합본 → 중복 제거

**선행 조건:** STACK 컴포넌트 데이터, Google Drive 구조

### 2-3. DS-knowledge-base `P0`

**목표**
디자인 시스템의 Knowledge Layer를 구축하여, LLM이 "왜 이 컴포넌트를 이렇게 써야 하는지"를 이해하고 올바른 UI를 생성할 수 있게 한다.

**현재 문제**
- 디자인 시스템 컴포넌트는 있지만 "사용 규칙"은 디자이너 머릿속에만 존재
- LLM에게 컴포넌트만 주면 잘못된 조합/배치의 UI를 생성
- 신규 입사자가 컨벤션을 익히는 데 오래 걸림

**솔루션**
- 컴포넌트별 사용 규칙을 기계 읽기 가능한 구조화 문서(MD/JSON)로 생성
  - "Primary 버튼은 섹션당 1개만"
  - "최소 터치 영역: 44px"
  - "명암비 4.5:1 이상"
- 접근성 기준, 디자인 의도, 맥락까지 포함
- 기존 디자이너 인터뷰/문서에서 Agent가 규칙 자동 추출

**솔루션 타입:** `Skill · 규칙 문서화` `Agent · 지식 추출`

**임팩트**
- LLM의 UI 생성 정확도 향상 (잘못된 컴포넌트 조합 감소)
- 디자인 리뷰에서 "이건 규칙 위반" 피드백 자동화 가능

**기대효과**
- Design-draft의 결론안 품질이 비약적으로 상승
- 디자인 시스템이 단순 라이브러리 → 지식 베이스로 진화
- 신규 입사자 온보딩 시 컨벤션 학습 시간 단축

**선행 조건:** 기존 사용 규칙/제약 조건 수집 (디자이너 인터뷰)

### 2-4. Figma-to-HTML-guide `P0`

**목표**
Figma 파일을 HTML로 정확하게 변환하기 위한 규칙 가이드 MD를 자동 생성하여, PO/Eng이 디자인을 바로 코드로 옮길 수 있게 한다.

**현재 문제**
- Figma → HTML 변환 시 Auto Layout과 Flexbox 매핑이 일관되지 않음
- 디자인 토큰이 CSS 변수로 어떻게 대응되는지 암묵지에 의존
- PO/Eng이 디자인 파일을 보고 HTML로 만들 때 해석 차이 발생

**솔루션**
- Figma 파일 구조를 분석하여 변환 규칙 MD 자동 생성:
  - Auto Layout → Flexbox/Grid 매핑 규칙
  - 디자인 토큰 → CSS 변수 변환 테이블
  - ODS 컴포넌트 → 시맨틱 HTML 태그 매핑
  - 간격/패딩/마진 규칙
- 파일별 맞춤 가이드 생성 가능

**솔루션 타입:** `Skill · 변환 가이드 MD` `Figma · 구조 분석`

**임팩트**
- Figma→HTML 변환 오류 감소
- PO/Eng의 디자인 해석 시간 단축

**기대효과**
- 디자이너가 만든 Figma 파일을 누구나 정확한 HTML로 변환 가능
- Design-draft에서 생성된 HTML의 품질 기준 역할
- DS-knowledge-base와 세트로 운영 → 변환 + 규칙 = 완전한 가이드

**선행 조건:** DS-knowledge-base와 연계

### 2-5. DS-logic-builder `P1`

**목표**
디자인 시스템의 Logic Layer를 정비하여, LLM이 컴포넌트의 구조와 동작 방식을 정확히 파악할 수 있게 한다.

**현재 문제**
- Figma 레이어 이름이 "Frame 124", "Rectangle 23" 등 의미 없는 상태
- 컴포넌트 속성(Variant/Size/State)이 일관되지 않게 정의됨
- Auto Layout 규칙이 파일마다 다름
- LLM이 Figma 구조를 읽어도 의미를 파악 불가

**솔루션**
- 네이밍 린팅: Category/Component/Variant/State 규칙 자동 검사 및 수정 제안
- 레이어 이름 정비: "Frame 124" → "Button Container" 자동 변환
- 컴포넌트 속성 정규화: Variant/Size/State 일관성 검증
- Auto Layout · 토큰 매핑 검증

**솔루션 타입:** `Figma Plugin · 린팅` `Skill · 속성 정규화`

**임팩트**
- LLM의 Figma 구조 이해도 향상
- 디자인 시스템 일관성 강화

**기대효과**
- AI가 컴포넌트의 계열과 변형을 자동 인식
- 디자인 시스템 품질 자체가 올라감 (AI 관계 없이도 가치)
- 블로그 참조: "AI가 이해할 수 있는 피그마 디자인 시스템 설계하기" 10가지 원칙 중 #1, #3, #4, #8 적용

**선행 조건:** ODS 컴포넌트 속성/네이밍 현황 파악

### 2-6. DS-code-bridge `P1`

**목표**
디자인 토큰과 코드를 동기화하여, 디자인 변경이 자동으로 코드에 반영되는 파이프라인을 구축한다.

**현재 문제**
- 디자인 토큰 변경 시 개발에 수동으로 전달해야 함
- Figma 컴포넌트와 프론트엔드 코드 간 매핑이 암묵적
- 토큰 값이 코드에 하드코딩된 경우 존재

**솔루션**
- 토큰을 JSON/코드로 자동 내보내기 (Style Dictionary 등 활용)
- Figma 컴포넌트 ↔ 프론트엔드 코드 매핑 동기화
- Figma Code Connect 활용한 실시간 연동
- 변경 감지 → 자동 PR 생성

**솔루션 타입:** `MCP · Token Sync` `Figma · Code Connect`

**임팩트**
- 토큰 전달 커뮤니케이션 비용 제거
- 디자인-코드 간 불일치 제로화

**기대효과**
- LLM이 디자인/코드 양쪽을 동일한 구조로 이해
- 디자인 변경 → 코드 반영 자동화
- 블로그 참조: 원칙 #2(토큰), #7(매핑), #10(코드 연결) 적용

**선행 조건:** 토큰 JSON 스키마 정의, Figma Code Connect 세팅

### 3-1. Responsive-converter `P0`

**목표**
디자인 시안의 반응형 레이아웃 변환을 자동화하여, PC/모바일/태블릿 대응 작업 시간을 줄인다.

**현재 문제**
- PC 시안 완성 후 모바일/태블릿 별도 작업 필요 → 해상도당 추가 작업 발생
- 브레이크포인트별 레이아웃 재조정이 반복 노동
- 해상도별 폰트 스케일, 간격 조정이 일관되지 않음

**솔루션**
- 디자인 시스템 브레이크포인트 기준으로 자동 변환
- PC → 태블릿 → 모바일 레이아웃 자동 생성

### 5-1. Spec-policy-handoff `P0`

**현재 문제**

**솔루션**
- 디자인 파일 + 기획 문서를 분석하여 구조화된 스펙 문서 자동 생성:
  - 비즈니스 규칙 · 정책 (예: 쿠폰 적용 조건, 배송 정책)
  - 엣지 케이스 목록 (예: 재고 0, 할인 중복, 최대 수량 초과)
  - 상태별 분기 조건
  - API 연동 포인트
- 개발자가 "왜 이렇게?"를 묻지 않아도 되는 수준

**솔루션 타입:** `Skill · 정책 스펙 문서` `Agent · 규칙 추출`

**임팩트**
- 핸드오프 후 질문 왕복 횟수: 평균 5~10회 → 1~2회
- 개발 착수 지연 감소

**기대효과**
- 셀라: 디자인 handoff보다 정책 handoff가 더 중요 — 이 니즈를 직접 해결
- 디어: Eng-handoff와 연계하여 정책+구현 스펙을 동시 전달
- 핸드오프 품질의 근본적 개선

**선행 조건:** 정책 문서 구조 정의

### P-1. Skill-hub `P0`

- "그 스킬 어떻게 쓰는 거야?" 질문 반복
- 스킬 중복 제작, 버전 관리 안 됨

**솔루션**
- Git 기반 스킬 저장소 운영
- 스킬 등록 · 검색 · 버전 관리
- 제작자, 프롬프트 · MCP 조합, 사용법 가이드 포함
- `/skill` 커맨드로 바로 사용 가능한 구조

**솔루션 타입:** `Git Repo · 스킬 저장소` `Skill · 레지스트리`

**임팩트**
- 스킬 재사용률 향상
- 중복 제작 방지

**기대효과**
- 팀 AI 역량이 공유 자산으로 축적
- 새 팀원도 기존 스킬 바로 활용 가능

**선행 조건:** Git 저장소 구조 설계, 스킬 템플릿 정의

### P-2. Design-AI-tips `P0`

**목표**
디자인 AI 활용 팁 · 노하우 · 프롬프트 레시피를 모아놓는 팀 지식 허브를 구축한다.

**현재 문제**
- AI 활용 노하우가 개인에게만 축적
- "이거 어떻게 시켰더니 잘 되더라"가 공유 안 됨
- 같은 시행착오를 팀원마다 반복

**솔루션**
- Notion 기반 AI 팁 지식 허브
- "이럴 때 이렇게 쓰면 잘 된다" 축적
- 태그 · 카테고리별 검색 가능
- Skill로 연동하여 `/ai-tips` 검색

**솔루션 타입:** `Skill · 팁 검색` `MCP · Notion`

**임팩트**
- AI 활용 학습 곡선 단축
- 팀 전체 AI 활용 수준 상향 평준화

**기대효과**
- 한 사람의 발견이 팀 전체 자산
- 신규 입사자 AI 온보딩 가속

**선행 조건:** Notion 지식 허브 구조 설계

### P-3. Insight-archive `P0`

**목표**
팀의 실험 결과 · 지표 하이라이트 · 도메인 지식을 구조화하여, 누구든 AI를 통해 맥락과 함께 검색할 수 있게 한다.

**현재 문제**
- 과거 실험 결과, 의사결정 근거가 Notion/Slack에 흩어져 있음
- 신규 입사자가 도메인 맥락을 파악하는 데 수개월 소요
- "왜 이런 결정을 했는지" 히스토리가 사라짐

**솔루션**
- Notion · Slack 히스토리에서 인사이트 자동 수집
- 실험 결과, 지표 하이라이트, 핵심 의사결정 구조화
- AI가 맥락과 함께 답변하는 검색 가능 지식 베이스
- 주기적 자동 업데이트

**솔루션 타입:** `Agent · 지식 수집` `MCP · Notion` `MCP · Slack`

**임팩트**
- 도메인 맥락 파악 시간: 수개월 → 1~2주
- 의사결정 근거 검색: 수시간 → 즉시

**기대효과**
- "3년차가 1주일 만에 가질 수 있는 도메인 맥락"
- 조직 변경/이동 시 맥락 전달 비용 최소화
- 과거 실험 재반복 방지

**선행 조건:** Notion/Slack 히스토리 접근, 지식 분류 체계 설계

### P-4. MCP-connector-hub `P1`

**목표**

**현재 문제**
- Athena, Redash, Slack, Notion 등 MCP 설정을 개인별로 해야 함
- 새 팀원 합류 시 MCP 세팅에 시간 소요
- 설정 버전이 팀원마다 다름

**솔루션**
- 팀 레벨 `.mcp.json` 공유 설정 관리
- 새 팀원 합류 시 원클릭 동기화
- MCP 추가/변경 시 팀 전체 자동 반영

**솔루션 타입:** `MCP · 공유 설정` `Config · .mcp.json`

**임팩트**
- MCP 세팅 시간: 1~2h → 5분
- 설정 불일치 제거

**기대효과**
- 새 팀원 온보딩 마찰 최소화
- 모든 Skill/Agent가 동일한 MCP 환경에서 동작 보장

**선행 조건:** 팀 공통 MCP 목록 정리
