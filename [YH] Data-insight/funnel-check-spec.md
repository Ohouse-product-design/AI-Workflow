# funnel-check 스킬 관리 문서

> 이 문서는 `/funnel-check` 스킬의 **설계 의도와 히스토리**를 관리한다. 스킬 파일 자체는 런타임 지시문(how)이고, 이 문서는 그 배경(why)이다. 스킬을 수정하기 전에 이 문서부터 읽어라.

## 개요

| 항목 | 내용 |
|------|------|
| 스킬 파일 | `~/.claude/commands/funnel-check.md` |
| 호출 | `/funnel-check [플로우명 또는 자연어]` |
| 목적 | 프로덕트 디자이너가 자기 오너십 화면의 플로우 전환률을 정확한 데이터로 측정 |
| 버전 | v1 (MVP) |
| 최초 작성 | 2026-04-10 |
| 마지막 갱신 | 2026-04-10 |

## 왜 만들었나 (문제 정의)

### 기존 방식의 문제
기존에 `athena mcp`로 "X 화면의 전환률 알려줘"라고 요청하면, 매번 다른 결과가 나왔다. 원인은:

1. **page_id 해석의 모호함** — "상품 상세"라는 말이 PDP인지 PDP_STYLINGSHOT인지, 관련 모든 PDP_*인지 요청자마다 다르게 해석
2. **플로우 시작/끝의 자의적 선정** — "어디서부터 어디까지가 전환"인지가 쿼리 작성자 판단에 달림
3. **object 필터의 부정확** — 버튼 클릭을 `object_section_id` / `object_type` 어떻게 잡을지가 매번 달라짐

데이터는 정확성이 생명이고, 정확성이 흔들리면 분석 결과가 설득력을 잃는다.

### 해결 방향
AI가 추측으로 쿼리를 짜지 말고, **사용자가 직접 폰으로 걸어본 로그**를 정답지로 삼는다. 실제 로그에 찍힌 page_id, object_section_id, object_type을 기반으로 플로우 정의를 고정한 뒤 쿼리를 만든다.

그리고 **자동 실행하지 않는다.** 온보딩 스킬처럼 한 단계씩 물어가며 사용자와 함께 정답을 만든다.

## 핵심 원칙 (5개)

1. **자동 실행 금지.** 각 단계에서 사용자 확정 후에만 다음으로 진행.
2. **자의적 플로우 연결 금지.** 로그에서 후보만 뽑아 제시, 연결은 사용자가 한다. AI가 스스로 연결하면 접합성이 떨어진다 — 이게 이 스킬의 존재 이유.
3. **쿼리는 전문을 보여주고 승인 후 실행.** 쿼리가 정답지이므로 사람 눈으로 검증 가능해야 한다.
4. **Detour 처리는 2층 분리.**
   - 층 1 (내 로그 복원): 뒤로가기/반복/잘못 진입을 정규화로 걸러 후보만 제시. 연결은 사용자.
   - 층 2 (전환률 쿼리): 기본 느슨한 퍼널. 중간에 다른 페이지가 껴도 도달하면 전환으로 본다.
5. **MD + HTML 자동 동기화.** Flow Definition은 MD로 저장하고 매 갱신마다 HTML 재생성.

## 워크플로우 요약

| Step | 이름 | 주요 작업 | 결과물 |
|------|------|----------|--------|
| 0 | 기존 정의 확인 | `./flows/{플로우명}.md` 존재 확인, 이어서/새로 시작 선택 | — |
| 1 | 맥락 수집 | 도메인, 전환 목표, 측정 기간 하나씩 물어보기 | MD §1 |
| 2 | 내 로그 그라운딩 | user_id + 좁은 시간 구간 → PAGEVIEW 추출 → 정규화 → 사용자 확정 | MD §2 |
| 3 | 화면 요소 확정 | CLICK 추출 + `get_page_spec` 교차검증 + trigger 사용자 확정 | MD §3 |
| 4 | Athena 쿼리 | 느슨한 퍼널 템플릿 → 승인 → 실행 → 결과 기록 | MD §4 |
| 5 | HTML 시각화 | omtm_review 스타일 다크모드 HTML 재생성 | `.html` |

## 파일 구조

```
~/.claude/commands/funnel-check.md    ← 스킬 본체 (런타임 지시문)
./flows/
  ├── {플로우명}.md                     ← Flow Definition + 쿼리 + 결과
  └── {플로우명}.html                   ← 매 MD 갱신마다 자동 재생성
```

## 플로우명 규칙

형식: `{domain}-{from_page}-to-{to_page}[-{variant}]`

- kebab-case, 소문자
- `{domain}`: `commerce` / `content` / `o2o` / `common`
- `{from_page}` / `{to_page}`: 첫/마지막 page_id (소문자, underscore 유지)
- `{variant}`: 같은 start/end 변형이 있을 때만

예: `commerce-pdp-to-cart`, `content-clp_project-to-cdp_project`

## 결정 로그

설계 과정에서 내린 결정과 그 이유. 나중에 "왜 이렇게 됐지?" 싶을 때 참고.

### 2026-04-10 최초 설계

**① "내 로그" 소스: user_id + 좁은 시간 구간**
- 대안: 세션 ID 기반 후보 여러 개 제시 → 나중에 "시간을 잘 모르겠다" 케이스 나오면 추가
- 결정 이유: 노이즈 최소화가 정확성의 핵심. 사용자가 언제 테스트했는지 기억한다는 가정.

**② Flow Definition 저장: MD 파일**
- 대안: JSON, Notion, Confluence
- 결정 이유: 사용자가 직접 열어서 읽고 수정할 수 있어야 함. 버전관리 친화.

**③ 런타임 MCP: athena mcp only**
- 대안: redash mcp 병용
- 결정 이유: 단일 실행 경로가 디버깅/재현성에 유리. redash는 설계 단계 참조용.

**④ 세그먼트 정의: MVP에서 제외**
- 당초 계획: redash에서 신규/활성/부활 표준 SQL을 찾아 스킬에 상수로 박기
- 조사 결과: 전사 표준 SQL 정의가 **존재하지 않음**. 각 팀(BA, CRM, 글린다, 집페이)이 자체 정의 사용. `ba_preserved.user_seg_rfd_v2`가 BA팀 유지 공용 마트로 확인되나 스키마(세그먼트 라벨 컬럼 유무) 미확인.
- 결정: MVP에서는 전체 유저 대상으로만. 세그먼트는 v2로 미룸 (로드맵 참조).

**⑤ Detour 처리: 2층 분리, 층 1은 정규화 후 사용자 확정**
- 초기안: 층 1 정규화 후 AI가 자동으로 플로우 연결
- 수정: AI가 자의적으로 연결하면 또 다른 오염원이 됨. 정규화 결과를 후보로 제시하고 연결은 사용자가 확정.
- 층 2 (쿼리): 느슨한 퍼널 기본, 엄격 퍼널은 옵션.

**⑥ 시각화: Figma 대신 HTML**
- 초기안: Figma MCP의 `generate_diagram`
- 수정: HTML로 충분. MD가 갱신될 때마다 자동 재생성.
- 스타일 레퍼런스: [omtm_review_260302](https://static-contents.datapl.datahou.se/v2/contents/omtm/omtm_review_260302.html) (다크모드, Recharts, 퍼널 카드)
- 보조 레퍼런스: [product_attribute_coverage](https://static-contents.datapl.datahou.se/v2/contents/owen_test/product_attribute_coverage.html) (KPI 카드 레이아웃)

**⑦ 스킬 파일 위치: `~/.claude/commands/`**
- 기존 log-* 스킬과 동일한 위치. slash command로 호출 가능.

**⑧ 플로우명 규칙: `{domain}-{from_page}-to-{to_page}[-{variant}]`**
- Step 1 끝에서 자동 생성 → 사용자 확정 → 파일 생성.

## 알려진 제한 / 미해결 질문

- **세그먼트 미지원**: 신규/활성/부활별 전환률 분석 불가. v2에서 해결 예정.
- **HTML 재생성 방식**: 매번 전체 HTML 재작성. 플로우가 커지거나 결과가 많아지면 느려질 수 있음. 성능 이슈 발생 시 섹션별 부분 갱신으로 리팩터.
- **다중 플로우 비교 미지원**: 두 플로우(예: PDP→CART vs PDP_STYLINGSHOT→CART)를 한 화면에서 비교 불가. v2 후보.
- **`ba_preserved.user_seg_rfd_v2` 스키마 미확인**: 세그먼트 라벨 컬럼 존재 여부 모름. 세그먼트 v2 설계 전에 반드시 확인 필요. 확인 방법 — BA팀 Sophie Cho 문의 또는 athena `DESCRIBE` 쿼리.
- **시간대 처리**: 사용자가 말하는 "오후 2시쯤"이 KST인지 UTC인지 모호. 스킬이 KST로 가정하고 변환하는지 명시 필요.
- **"내 로그"를 못 찾는 케이스**: 사용자가 테스트 시각을 잘못 기억하거나, 해당 기간에 로그가 없으면 그라운딩 실패. 폴백 플로우 없음.

## 로드맵

### v2 후보 (우선순위 순)

1. **세그먼트 지원**
   - 선결 과제: `ba_preserved.user_seg_rfd_v2` 스키마 확인
   - 스키마에 라벨 컬럼이 있으면: 런타임에 LEFT JOIN으로 세그먼트 분리
   - 없으면: BA팀과 정의 합의 후 `recent_days` 임계값 상수화
2. **다중 플로우 비교**
   - 한 HTML에 두 개 이상 플로우를 side-by-side로 렌더
   - MD 포맷 확장 필요 (`## 2a`, `## 2b` 같은 식)
3. **세션 후보 선택 기능**
   - 사용자가 테스트 시각을 모를 때, 최근 N일치 세션 요약을 보여주고 고르게 함
4. **엄격 퍼널의 명시적 옵션화**
   - 현재는 Step 4 중에 물어보는 식. 스킬 호출 시점에 `--strict` 같은 플래그로 받게 개선

### 유지보수 체크포인트

- **분기마다**: 실제로 몇 번 썼는지, 어디서 막혔는지 돌아보기
- **로그 스키마 변경 시**: `log.analyst_log_table` 스키마 변경이나 새 enum 추가되면 스킬 쿼리 템플릿 갱신 필요
- **log-query 스킬 변경 시**: funnel-check는 log-query의 규칙(파티션, 필터)을 승계하므로 상호 체크
